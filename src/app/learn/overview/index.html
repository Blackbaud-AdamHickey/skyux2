SKY UX 2 Overview

The SKY UX framework provides a consistent, cohesive user experience for Blackbaud products and for third-party developers. SKY UX 2 continues to provide the components, services, and modules that allow developers to incorporate Blackbaud styles and patterns. In addition, SKY UX 2 also provides a developer tool called the SKY UX Builder to abstract many of the complexities of modern web development. And SKY UX 2 also includes a continuous integration layer for testing and a hosting solution for Blackbaud applications.

SKY UX 2 enables developers to focus on features instead of infrastructure. It creates user experience interactions and allows feature engineers to consume them.

Since Angular 2 was a complete rewrite of AngularJS, we used it as an opportunity to expand SKY UX. SKY UX Builder takes the application building aspects of Angular 2 and wraps them up. We still expect developers to write JavaScript/Typescript and build features, but we don’t expect them to do all the underlying work such as handling Webpack. Instead, SKY UX Builder abstracts that layer from developers.

SKY UX vs. SKY UX 2

SKY UX 2 continues to provide many of the same benefits as the first version of SKY UX. It continues to provide design patterns that implement a consistent, cohesive user experience along with guidance on how to apply components and the context for when to use them. And it continues to provide Angular components, services, and modules that allow developers to create user interfaces that incorporate SKY UX styles and patterns.

In addition, SKY UX 2 includes several improvements and new capabilities, including the following benefits:

  • SKY UX 2 takes advantage of Angular 2 to keep SKY UX modern while increasing the tooling, testing, and performance available.

  • SKY UX Builder provides for a rapid development process with minimal developer setup. It creates single-page applications and handles details such as transpilation, linting, unit testing, and bundling/chunking/minifying.

  • A continuous integration layer runs unit tests, visual regression tests, and end-to-end tests in multiple platforms and browsers when a SPA's code changes or when an SPA dependency releases a new version. This layer also stages the SPA for additional testing and validation.

  • SKY UX Host acts as a gateway for all requests to SKY UX SPAs. When serving locally, it pulls files locally. And for production, it pulls files from a registry that stores static files when SKY UX Builder runs in the continuous integration layer.

SKY UX Builder

SKY UX Builder is cross-platform command line developer tool to create, test, and run SPAs locally. It abstracts most of the tedious complexities of modern web development, including bundling configuration, unit test configuration, and routing/module configuration. This allows developers to focus on features, not infrastructure. And it allows SPAs to be developed, tested, deployed, and hosted in a consistent way across an entire application.

SKY UX Builder provides for a rapid development process with minimal developer setup. It handles details such as transpilation, linting, unit testing, and bundling/chunking/minifying. In addition, SKY UX Builder also handles the following details:

  • By default, SKY UX Builder uses automatic route configuration based on the project's folder structure. This means that you must place files in specific locations to make sure that they are recognized. For example, to create a page, you add a folder in the existing \src\app folder and add an index.html file to that new folder.

  • Provides build and testing configurations. This includes settings to use Webpack to handle minification and build JavaScript assets, Karma unit tests, and Protractor end-to-end tests.

  • Allows easy implementation of Blackbaud Authentication.

Continuous integration layer

The continuous integration layer of SKY UX runs unit tests, visual regression tests, and e2e tests in multiple platforms and browsers when the SPA code changes or when an SPA dependency releases a new version. It also handles staging the SPA for additional testing and validation.

SKY UX Host

The SKY UX Registry serves as the mechanism where the status of builds and SPAs is shared from the continuous integration layer to SKY UX Host. When running SKY UX Builder in the continuous integration layer, the static files that it produces (JavaScript files, JSON files, etc.) are stored in the SKY UX Registry, and the SKY UX Host looks for them there to serve them when the files are requested.

SKY UX Host acts as a gateway for all requests to SKY UX SPAs. When serving locally, it pulls files locally. And for production, it pulls files from a registry that stores static files when SKY UX Builder runs in the continuous integration layer.

In addition, SKY UX Host also creates a web service to handle the following details:

  • Serve up an HTML page that hosts the SPA.

  • Set the appropriate cache headers for the HTML page and static files.

  • Set the appropriate handlers to support HTML 5-style routing.

  • Serve as an origin for a content delivery network (CDN).

  • Register JavaScript and CSS for the SPA. This includes registering locale files based on browser culture, using sub-resource integrity checking for CDN-hosted files, and using a fallback strategy for when the CDN is unavailable.

  • Implement HTTP Strict Transport Security (HSTS).

  • Obtain, configure, and deploy SSL certificates.

  • Maintain highly availability through load-balancing, georeplication, and monitoring.
